how credits work, in words

A new mind gets a wallet at creation. Its Cognitive Credits = wallet balance of supported tokens: 1 MENTE = 1 credit; MOCA counts by USD value (USDC support planned). Holding MOCA with no MENTE triggers a small auto-swap to MENTE (the actionable credit). Each user request makes the mind pay MENTE to the collector — gasless for the user: the gas funder first sends a sliver of ETH to the mind wallet in the same moment (visible on-chain as paired transactions). Value leaves the loop via an automatic contract burn (~1.2% lifetime, being confirmed) and enters via Stripe purchases and manual treasury funding. Economy eras: the first generation (≤ Apr 2026) ran on the SWARM token (Ethoswarm) through the same collector hub; SWARM→MENTE migration ~Apr 8–24, 2026 — this dashboard covers the MENTE/MOCA era (SWARM-era integration: see Data gaps). ⚠ Note: a lookalike address 0xD8506866…F111 mimics the collector via zero-value transfers (address poisoning) — never copy the collector address from a transaction history.

Cognition consumption — minds spending MENTE

On-chain demand side of the economy: MENTE flowing from mind wallets into the Cognition Credits collection wallet (0xd85096…f111) when minds spend credits on cognition. Collected tokens recycle back into this treasury. Treasury-originated transfers to the collector are excluded; spenders are restricted to wallets that previously received treasury payouts. USD at that day’s token price. Early sample — treat consumption vs distribution parity as an early signal until the 30-day series completes.

Who pays for cognition? — treasury credits vs user-brought tokens

Anyone can send SWARM / MENTE / MOCA straight to a mind's wallet as spendable Cognition Credits — a real, on-chain revenue stream separate from Stripe. We detect it by conservation per mind wallet: if a mind spent more on cognition than this treasury ever credited it, the excess had to come from tokens the user supplied.

Method & caveats: for each mind, user-funded = max(0, consumed − treasury-credited), assuming treasury credits are spent first (so the total is a strict lower bound). All USD at each day's token price. "User-funded" here also includes mind→mind transfers (e.g. one mind awakening another), so it is not purely external deposits — separating those needs the wallet↔mind map (see Data gaps). Stripe top-ups are counted as treasury-funded (they leave from this treasury), so this figure is additional to Stripe revenue.

AI-inferred interpretation

Everything in this amber zone is inferred — from transfer sizes, timing heuristics, or projections — not read from a ledger. Each block states its method. Treat as triage, not truth.
AI-inferredPayout type classification — invokes, equips, credits, Stripe

Method: each transfer is snapped to the platform's known price points ($0.10 invoke, $1 equip, $3 credit, $5 referral, $10/25/50 Stripe) by its USD value at that day's token price, ±8% — the same tolerance as the factual size bands above, so band counts and type counts always agree. This is inference from size, not ground truth: a per-transfer memo from the payout contract (requested) would make it exact. The $3 bucket mixes new-user credits and internal top-ups — the internal top-ups sheet is the authoritative split. Stripe-sized transfers are deliveries of fiat purchases (revenue-backed); confirming them requires the Stripe ledger, which this page does not read yet.

AI-inferredCreator earnings ranking (classified invokes + equips)

Method: recipients of transfers classified as invoke/equip, ranked by USD, share of ALL classified creator earnings. Inherits the size classification's uncertainty.

AI-inferredPayout pattern monitor

Method: we look at each earner's payout timing and ask: does it look human-irregular or robotically regular? Three signals: timing variety, repeating rhythm, and rapid-fire bursts (technical: gap entropy, lag-1 autocorrelation, 60-second burst share). Cut-offs are judgment calls; flags mean "worth a look", never "confirmed abuse". Automated invocation can be legitimate — the automation policy is an open decision. Flagged rows sort first.
